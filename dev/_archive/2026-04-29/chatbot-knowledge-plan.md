## 0. 한 줄 요약 (바이브 코더용 비유)

> 지금 티즈는 **"매장 카탈로그 앞장 몇 줄만 암기한 신입 점원"**이에요.
> 리뉴얼 후 티즈는 **"매장 전체 상품·가격표·규칙집을 요약한 핸드북을 항상 손에 쥐고 있는 베테랑 점원"**이 됩니다.
> **핸드북(정적 지식) + 매장 전산(실시간 DB)** 이 두 도구를 함께 쓰는 방식이 핵심입니다.

## 1. 현재 상태 진단 — Phase 1 한계점

### 1-1. 지금 티즈가 알고 있는 것
Phase 1 MVP에서 티즈는 아래 방식으로만 "지식"을 얻습니다.

| 경로 | 내용 | 한계 |
|------|------|------|
| 시스템 프롬프트 하드코딩 (`ai.js` L227~244) | 회사 소개, 최소수량, 할인율, 제작기간, 배송비, 반품 7일 | 정적, 코드 수정 없인 업데이트 불가. 누락 정보 다수 |
| 상품 키워드 감지 → DB 조회 (`PRODUCT_KEYWORDS`, L150) | 질문에 상품 키워드 있으면 3건만 LIKE 검색 → 프롬프트 주입 | 키워드 매칭 조잡, 자연어 질문 실패, 가격/옵션/원단 누락 |
| 클라이언트 규칙 분기 (`chatbot.js` routeIntent) | 농구/축구/배구/야구 키워드, "배송비/교환" FAQ | 4종목/FAQ 5~6개만 커버. 나머지는 전부 Gemini 폴백 |

### 1-2. 구체적 실패 시나리오 (실제 질문 예시)

사용자가 이렇게 물을 때 **티즈는 틀리거나 모른다고 답합니다**:

- "XL 사이즈도 있어요?" → DB에 `sizes` 컬럼(예: `'S~3XL'`) 있지만 프롬프트에 미주입 → 추측 답변 위험
- "승화전사랑 나염 차이가 뭐예요?" → 어디에도 설명 없음 → 환각 가능
- "학교스포츠클럽은 얼마 할인돼요?" → DB에 `clubPrice` 컬럼 있지만 프롬프트에 없음
- "페가수스 어센틱 홈 저지 사고 싶어요" → 상품명 정확 매칭은 되지만, 옵션/재고/이미지 URL은 미주입
- "어제 이벤트 공지 있었죠?" → board(공지) 데이터 0건 주입됨
- "시안 수정은 몇 회까지 돼요?" → 어디에도 없음. about.html에 5단계 프로세스 설명만 존재

**핵심 문제**: 사이트에 "데이터는 있지만 티즈가 못 보는" 상태.

## 3. 접근법 A/B/C 비교 (핵심)

### 접근법 A. 확장 컨텍스트 주입 (Phase 1 확장형)
**비유: 손님 질문 들을 때마다 관련 카탈로그 페이지를 그때그때 뜯어서 직원한테 넘겨주는 방식**

- 방식: 키워드 매칭 규칙을 늘려서 상품/정책/FAQ별로 DB/JSON에서 조회 → 프롬프트에 주입
- 예: "사이즈" 키워드 → 상품 sizes 컬럼 포함 / "배송" → 배송 FAQ 블록
- 구현: `ai.js`의 `buildProductContext`를 `buildKnowledgeContext`로 확장, 의도별 분기

| 항목 | 평가 |
|------|------|
| 예상 구현 시간 | **4~6시간** (Phase 1 코드 재활용) |
| 월 예상 비용 | 현재 수준 유지 (추가 토큰 30% 증가 수준) |
| 운영 복잡도 | **2/5** (매우 단순) |
| 답변 정확도 | **3/5** (키워드 매칭 한계) |
| 인프라 추가 | 없음 |
| STIZ 적합도 | 중상 — MVP엔 최적, 확장성은 낮음 |

**장점**: 즉시 적용 가능, 실시간 데이터, 추가 비용·인프라 0
**단점**: 자연어 질문 매칭 약함("저렴한 농구복" 같은 모호한 표현에 약함), 키워드 규칙이 수십 개로 늘면 유지보수 지옥

### 접근법 B. 정적 지식베이스 JSON (사전 빌드)
**비유: 매장 규정집을 한 권의 깔끔한 핸드북으로 만들어 직원이 항상 허리춤에 차고 다니는 방식**

- 방식:
  1. 배치 스크립트가 DB + HTML + 가격표를 읽어 `server/data/knowledge/knowledge.json` 1개 파일 생성
  2. 서버 시작 시 메모리에 로드 (18MB JSON도 Node 메모리에 즉시 올림)
  3. 의도별로 "상품 섹션" / "정책 섹션" / "FAQ 섹션"을 선택해 프롬프트에 주입
  4. 상품 추가/수정 시 `npm run build-knowledge` 재실행 (또는 관리자 트리거)

- 구조 예시:
```json
{
  "company": { "name": "STIZ", "founded": 2016, ... },
  "policies": {
    "shipping": "5만원 이상 무료, 3,000원 배송비...",
    "refund": "수령 7일 이내, 커스텀 제외...",
    "bulk_discount": "10벌 5% / 20벌 10% / 50벌 15%"
  },
  "products_summary": [
    { "id": 1, "name": "페가수스 어센틱 홈", "category": "축구의류", "price": 45000, "sizes": "S~3XL", "fabric": "어센틱" },
    ... 271건
  ],
  "categories": [...],
  "faq": [
    { "q": "승화전사와 나염 차이", "a": "..." },
    { "q": "시안 수정 횟수", "a": "3회까지 무료" }
  ],
  "design_lab": { "2d": "...", "3d": "..." }
}
```

| 항목 | 평가 |
|------|------|
| 예상 구현 시간 | **10~14시간** (빌드 스크립트 + FAQ 작성 + 의도 분기) |
| 월 예상 비용 | 프롬프트 평균 길이 2~3KB 증가 → +20% 토큰, **약 $2~5/월 추가** |
| 운영 복잡도 | **3/5** (재빌드 타이밍 관리 필요) |
| 답변 정확도 | **4/5** (정책/FAQ 정확도 크게 상승) |
| 인프라 추가 | JSON 파일 1개, 빌드 스크립트 1개 |
| STIZ 적합도 | **상 — 271개 상품, 소규모 정책 문서 규모에 딱 맞음** |

**장점**: 관리 편함, 상품수 271개면 전체 요약도 프롬프트에 부분 주입 가능, FAQ 일원화
**단점**: 실시간 재고/주문 상태는 별도 DB 쿼리 필요 → A와 조합 필수, FAQ 수동 작성 부담

### 접근법 C. RAG (벡터 임베딩 + 유사도 검색)
**비유: 핸드북의 모든 문장에 "의미 좌표"를 찍어서, 손님 질문 좌표와 가까운 문장만 자동으로 뽑아 보여주는 방식**

- 방식:
  1. 전체 지식(상품 설명 + 정책 + FAQ)을 300~500자 **청크(chunk)**로 쪼갬 → 약 500~1000개 청크
  2. Gemini `text-embedding-004` API로 각 청크를 768차원 벡터로 변환 → `sqlite-vec` 또는 `server/data/embeddings.json`에 저장
  3. 사용자 질문도 같은 API로 임베딩 → 코사인 유사도 top-5 청크 검색 → 프롬프트 주입
  4. 상품/정책 변경 시 해당 청크만 재임베딩

**비유(상세)**: 임베딩 = 문장을 숫자 지도 좌표로 바꾸는 것. "축구 유니폼 가격"과 "사커 저지 얼마"는 다른 단어지만 지도상 거리가 가까워 같은 영역을 찾음.

| 항목 | 평가 |
|------|------|
| 예상 구현 시간 | **20~30시간** (청크 분할 + 임베딩 파이프라인 + 검색 로직 + 재임베딩 트리거) |
| 월 예상 비용 | 초기 임베딩 1회 약 $0.5 (500~1000 청크), 질문당 임베딩 $0.00001 × 사용량, **총 $3~10/월** |
| 운영 복잡도 | **5/5** (벡터 DB 관리, 청크 크기 튜닝, 재임베딩 자동화) |
| 답변 정확도 | **5/5** (자연어 질문 매칭 최상) |
| 인프라 추가 | sqlite-vec 네이티브 빌드 OR 인메모리 벡터 배열 + 선형 스캔 |
| STIZ 적합도 | **중 — 271개 상품 규모엔 과한 투자** |

**장점**: "저렴한 농구복" / "여름에 시원한 원단" 같은 자연어 쿼리에 강함
**단점**: 바이브 코더가 디버깅 어려움, 임베딩 API 추가 의존, 재인덱싱 자동화 필수, 규모 대비 투자과다

## 4. 하이브리드 분석 (A+B 권장)

**단독 접근법은 모두 약점이 있습니다**. STIZ 규모(271 상품 + 소규모 정책)에는 **A+B 하이브리드**가 최적입니다.

### 권장: B(정적 JSON 지식베이스) + A(실시간 DB 보조)

```
사용자 질문
    ↓
[의도 분류 — 규칙 기반 경량]
    ↓
┌───────────────────┬────────────────────┐
│ 정책/FAQ/회사정보  │ 상품/재고/가격      │
│   → B에서 조회     │   → A(실시간 DB)   │
│   knowledge.json   │   + B(상품 요약)    │
└───────┬───────────┴──────┬─────────────┘
        │                  │
        ↓                  ↓
   Gemini 프롬프트에 합쳐서 주입
        ↓
   답변 생성
```

**왜 B+A인가**:
- 정책·FAQ·회사정보: 변경 빈도 낮음 → JSON 캐시 최적 (B)
- 상품·가격·재고: 실시간성 중요 → DB 직조회 (A)
- 이 조합이 **정확도 4.5/5, 비용 최저, 복잡도 중간**

### C(RAG)는 Phase 3+ 검토 사항
- 상품이 1000개 이상 늘거나, "자연어로 애매하게 묻는" 케이스가 로그상 30% 이상 관찰되면 그때 추가
- 지금 271개면 **전체 상품명 + 카테고리를 1회 프롬프트(약 8KB)로 다 넣어도** Gemini 2.5-flash 컨텍스트(1M 토큰)엔 무해

## 5. 추천안 상세 아키텍처 (B+A 하이브리드)

### 5-1. 파일 구조
**비유: 건물에 "지식 도서관" 한 층을 새로 짓는 것**

```
server/
├── data/
│   └── knowledge/
│       ├── knowledge.json        ← 자동 빌드 산출물 (커밋 제외)
│       ├── faq.json              ← 사람이 직접 작성/관리
│       ├── policies.json         ← 사람이 직접 작성/관리
│       └── company.json          ← 회사 정보 (사람이 관리)
├── services/
│   └── knowledge.js              ← 지식 로드/검색/주입 (신규 모듈)
├── routes/
│   └── ai.js                     ← knowledge.js 사용하도록 수정
└── scripts/
    └── build-knowledge.js        ← 빌드 스크립트 (신규)
```

### 5-2. 데이터 수집 파이프라인

**빌드 시점 (npm run build-knowledge 실행 시)**:
```
[products 테이블] ─┐
[product_categories] ─┤
[product_reviews] ─┤  →  build-knowledge.js  →  knowledge.json
[faq.json (수동)] ─┤       (SQL 쿼리 + JSON 합치기 + 요약)
[policies.json] ─┤
[company.json] ─┘
```

**런타임 (서버 시작 시)**:
- `knowledge.js`가 `knowledge.json` 1회 로드 → 메모리 캐시
- `/api/chat` 요청마다 캐시에서 의도별 섹션 선택

### 5-3. 검색 로직 (의도 분류기)

**비유: 손님 질문을 듣고 "저건 정책 질문" / "저건 상품 질문" 라벨 붙이는 프런트데스크**

```javascript
// 의도 분류 규칙 (서버 knowledge.js)
const INTENTS = {
  policy: /(배송|교환|환불|반품|결제|수령|할인|쿠폰|무료배송)/i,
  product: /(상품|제품|유니폼|저지|바지|추천|인기|신상|가격|얼마)/i,
  size: /(사이즈|치수|XL|XXL|큰|작은|몸|키|몸무게)/i,
  custom: /(커스텀|제작|디자인|시안|로고|마킹|번호|이름|2D|3D)/i,
  category: /(축구|농구|배구|야구|종목|카테고리)/i,
  design: /(승화전사|나염|프린트|원단|소재|색상)/i,
  company: /(회사|소개|위치|주소|연락처|대표)/i
};

function detectIntents(message) {
  return Object.keys(INTENTS).filter(k => INTENTS[k].test(message));
}
```

감지된 의도별로 knowledge.json의 해당 섹션만 꺼내 프롬프트에 추가.

### 5-4. 프롬프트 엔지니어링

```
[시스템 프롬프트 - 티즈 페르소나 / 기본 규칙 - 약 800B]

[지식 컨텍스트 - 의도별 동적 주입 - 평균 1~3KB]
== STIZ 정책 ==
{policies 중 관련 항목만}

== 관련 상품 (DB 실시간) ==
{products 테이블에서 LIKE/카테고리 필터 top 5}

== FAQ ==
{faq.json 중 유사 질문 top 3}

[대화 히스토리 - 최근 4턴]

[현재 사용자 질문]
```

**중요**: 티즈가 **"모르는 건 모른다고 답하도록"** 프롬프트에 명시:
> "위에 제공된 정보에 없는 상품/가격/정책은 절대 추측하지 말고, '정확한 정보는 카카오톡 @stiz로 문의해주세요'라고 답변하세요."

### 5-5. 업데이트 주기

| 데이터 | 주기 | 방법 |
|--------|------|------|
| 상품 (products 테이블) | **실시간** (빌드 스킵, 쿼리 직접) | SQL |
| 카테고리 | 빌드 시 | SQL 스냅샷 |
| 공지 (board notice) | **실시간** | SQL 최신 3건 |
| 정책 (policies.json) | 수동 편집 시 + 서버 재시작 | 파일 수정 |
| FAQ (faq.json) | 수동 편집 시 + 서버 재시작 | 파일 수정 |
| 회사 정보 | 거의 없음 | 파일 수정 |

**Phase 3(선택)**: 관리자 페이지에서 FAQ 편집 → API 저장 → hot reload

## 7. 리스크 · 고려사항

### 7-1. 환각(Hallucination) — 가장 큰 위험
**비유: 티즈가 모르는 상품을 "있어요!"라고 지어내면 손님이 주문한 뒤 "그런 거 없는데요" 상황 발생**

**대응**:
- 시스템 프롬프트 최상단에 **금칙어 선언**:
  > "제공된 컨텍스트에 없는 상품명/가격/정책은 절대 지어내지 마세요. 모르면 '확인 후 안내드릴게요' 또는 '카카오톡 @stiz로 문의해주세요'로 답하세요."
- AI 응답 후 **검증 레이어**:
  - 응답에 숫자(가격)가 포함되면 → 해당 숫자가 컨텍스트에 실제 존재하는지 정규식 검사
  - 실제 없으면 → "정확한 가격은 상담원 확인이 필요해요" 자동 치환 (Phase 2-K2)
- UI에 **⚠️ AI 응답 뱃지** 30일간 노출 (소비자 신뢰 경고)

### 7-2. 개인정보/민감정보 노출
**비유: 손님 물건 맡긴 보관함 내용을 다른 손님에게 흘리면 안 됨**

**대응**:
- **orders / customers 테이블은 학습 대상에서 완전 제외**
- 주문 조회는 Phase 2(보류) 재개 시 별도 플로우로 처리 (본인 인증 → 직접 DB 조회 → Gemini 미경유)
- knowledge.json 빌드 시 **이메일/전화번호/주소 필드 자동 제거** 검증 루틴 포함

### 7-3. 가격 오답 → 법적/비즈니스 리스크
**비유: 매장 가격표를 잘못 읽은 직원 탓에 손님이 5만원 상품을 3만원으로 알고 주문하는 사고**

**대응**:
- 가격은 **반드시 DB 실시간 조회만** (JSON 캐시의 가격은 "참고용" 라벨)
- 할인/쿠폰 계산은 AI가 하지 않음 (규칙 기반 코드로 직접 계산)
- AI 응답에 가격이 포함되면 하단에 자동 추가: *"최종 가격은 장바구니에서 확인해주세요"*

### 7-4. 비용 폭증 시나리오
**비유: 전기요금 누진세 — 주 100명이 100번씩 호출하면 월 만원, 10배 늘면 월 10만원**

**현재 Gemini 2.5-flash 단가 (2026-04 기준 추정)**:
- 입력: $0.075 / 1M 토큰
- 출력: $0.30 / 1M 토큰
- 평균 질문당 4~6KB 프롬프트 + 1KB 응답 → 약 $0.0005

**월 사용량별 예상 비용**:
| 월 대화 수 | Phase 1 현재 | Phase 2-K2 도입 후 |

| 1,000건 | $0.5 | $1 |

**대응**:
- 키워드 기반 직답(FAQ 정답)은 **Gemini 호출 없이** 즉시 응답 → 호출 비율 30% 이하 유지

### 7-5. 데이터 신선도 (재고 없는 상품 추천)

## 10. 보류된 Phase 2(주문조회/쿠폰/공지)와의 관계

### 10-1. 지식 학습이 Phase 2를 얼마나 대체하나
| 기능 | 지식 학습으로 대체? | 판단 |
|------|-------------------|------|
| 비로그인 주문조회 | ❌ 불가 | 개인정보 보호 — 학습 대상 아님, 별도 플로우 필요 |
| 내 주문 조회 | ❌ 불가 | 로그인 사용자별 필터 — 학습 무관 |
| 쿠폰 코드 확인 | △ 부분 대체 | "어떤 쿠폰 있어요?" 같은 일반 질문은 가능. 코드 검증은 별도 API |
| 공지사항 조회 | ✅ 완전 대체 | "최근 공지" 실시간 DB 조회로 이미 커버 가능 |
| 세션 히스토리 localStorage | ❌ 불가 | UX 개선 영역, 지식 무관 |

### 10-2. 결론
**지식 학습(Phase 2-K) 완료 후, Phase 2 원래 범위를 축소 재검토 권장**:
- 공지 조회는 K1 + K2 과정에서 자연스럽게 커버됨
- 주문 조회는 별도 플로우(Phase 2-O)로 독립 기획
- 쿠폰은 Phase 2-K2에 "쿠폰 정보 FAQ" 수준으로 통합

### 10-3. 제안 실행 순서
```
[지금] Phase 1 MVP 완료
     ↓

     ↓

     ↓ (6개월 후 재판단)
Phase 2-K4 (선택) — RAG 전환
```
