## 현재 상태 요약

| 항목 | 상태 | 설명 |
|------|------|------|
| PortOne SDK | 로드됨 | checkout.html에 `iamport.js` CDN 포함 |

## Part 1: PG 결제 본격 연동 (PortOne)

### 1.1 개념 설명

**PortOne(구 아임포트)이란?**
- 비유: "결제 통합 중개소". 각 카드사/은행에 직접 계약하는 대신, PortOne 하나만 연결하면 여러 결제 수단을 한번에 쓸 수 있다.
- 실제 돈이 오가는 곳은 PG사(이니시스, NHN KCP 등). PortOne은 이 PG사들을 하나의 인터페이스로 묶어주는 역할.

**결제 흐름 (식당 비유)**:
```
1. 손님이 메뉴 선택 (장바구니 → 주문)
2. 서버가 "이 테이블 15만원"을 기록 (prepare API → 위변조 방지)
3. 손님이 카드 결제기에 카드 터치 (PortOne 결제창)
4. 카드사가 승인 (PG사 → PortOne → 우리 사이트)
5. 서버가 "진짜 15만원 결제됐나?" 확인 (complete API → 금액 검증)
6. 주문 확정
```

### 1.2 설정 절차 (외부 작업 — 코드 아님)

| 순서 | 작업 | 소요 시간 | 비고 |
|------|------|----------|------|
| 1 | [PortOne 관리자 콘솔](https://admin.portone.io) 회원가입 | 5분 | 사업자 등록번호 필요 |
| 2 | 채널 추가 → PG사 선택 | 10분 | 추천: KG이니시스 또는 NHN KCP |
| 3 | **테스트 모드**로 채널 활성화 | 즉시 | 테스트 키 자동 발급 |
| 4 | 가맹점 식별코드(merchantId) 확인 | 즉시 | 관리자 콘솔 → "내 식별코드" |
| 5 | REST API Key + Secret 발급 | 즉시 | 관리자 콘솔 → "결제 연동" → "인증키" |
| 6 | .env에 3개 키 입력 | 1분 | PORTONE_MERCHANT_ID, API_KEY, API_SECRET |
| 7 | 테스트 결제 진행 | 10분 | 실제 돈이 빠지지 않는 테스트 결제 |
| 8 | PG사 본계약 (프로덕션 전환) | 3~7일 | 사업자 서류 제출 → PG사 심사 |
| 9 | 테스트 키 → 라이브 키 교체 | 5분 | .env의 키만 변경 |

**주의**: 테스트 모드에서는 실제 결제가 일어나지 않는다. 프로덕션 전환 전까지는 안심하고 테스트 가능.

### 1.3 코드 수정 사항

현재 코드는 **거의 완성 상태**. 수정이 필요한 부분만 정리:

| 파일 | 수정 내용 | 이유 |
|------|----------|------|
| checkout.js (299행) | `pg: 'html5_inicis'` → PortOne 콘솔에서 발급된 채널키로 변경 | PG사 매칭 |
| checkout.js | 계좌이체/가상계좌 결제 옵션 추가 | 결제 수단 확장 |
| checkout.html | 계좌이체/가상계좌 라디오 버튼 UI 추가 | 결제 수단 선택 |
| payment.js | 가상계좌 입금 대기 상태 처리 + 웹훅 수신 | 가상계좌는 즉시 결제가 아님 |
| server.js | 웹훅 엔드포인트 등록 (`POST /api/payment/webhook`) | PortOne이 입금 확인을 알려줌 |

### 1.4 지원할 결제 수단

| 결제 수단 | 설명 | 구현 난이도 | 비고 |
|-----------|------|-----------|------|
| 신용/체크카드 | 가장 기본. 즉시 결제 | 이미 구현됨 | pg 값만 교체 |
| 실시간 계좌이체 | 은행 앱으로 바로 이체 | 낮음 | pay_method: 'trans' |
| 가상계좌 | 1회용 계좌 발급 → 입금 대기 | 중간 | 웹훅 필요 (입금 확인) |
| 무통장 입금 | 기존 방식 유지 | 완료 | 변경 없음 |

### 1.5 가상계좌 특이사항

가상계좌는 다른 결제와 다르다:
- 비유: "일회용 통장"을 만들어주고, 고객이 나중에 입금하면 그때 확인되는 구조
- 결제창에서 "완료"를 눌러도 아직 돈이 안 들어온 상태
- 고객이 실제로 입금하면 PortOne이 우리 서버에 **웹훅(알림)**을 보냄
- 그 웹훅을 받아서 주문 상태를 "입금 확인"으로 변경해야 함

```
[고객] → 가상계좌 발급 요청 → [PortOne] → 1회용 계좌번호 발급
[고객] → 은행 앱에서 해당 계좌로 이체
[은행] → [PG사] → [PortOne] → 우리 서버 웹훅 → 주문 상태 변경
```

### 1.6 테스트 → 프로덕션 전환

| 단계 | 테스트 모드 | 프로덕션 모드 |
|------|-----------|------------|
| API URL | 동일 (api.iamport.kr) | 동일 |
| 실제 결제 | X (가짜 결제) | O (진짜 결제) |
| 키 종류 | 테스트 키 | 라이브 키 |
| 전환 방법 | - | .env의 키 3개를 라이브 키로 교체 |
| PG사 심사 | 불필요 | 필요 (3~7영업일) |

## Part 2: 간편결제 연동

### 2.1 PortOne 통합 vs 직접 연동 비교

| 항목 | PortOne 통합 | 직접 연동 |
|------|------------|----------|
| 비유 | 백화점 종합 안내 데스크 | 각 매장 직접 방문 |
| 개발 양 | checkout.js의 pg 파라미터만 변경 | 각 간편결제별 SDK + 서버 코드 별도 |
| 추가 코드 | 거의 없음 (pg값만 다름) | 각각 수백 줄씩 |
| 수수료 | PortOne 수수료 + PG 수수료 | PG 수수료만 |
| 추천 | **강력 추천** | 트래픽 매우 클 때만 |

**결론: PortOne 통합 방식 추천**. 코드 한 줄(pg 파라미터)만 바꾸면 간편결제 추가 가능.

### 2.2 PortOne에서 지원하는 간편결제 목록

PortOne을 통해 다음 간편결제를 **코드 추가 없이** 사용 가능:

| 간편결제 | PortOne pg 값 | 별도 계약 필요 | 비고 |
|---------|--------------|-------------|------|
| **네이버페이** | `naverpay` | O (네이버페이 가맹점) | 네이버 쇼핑 연동 가능 |
| **카카오페이** | `kakaopay` | O (카카오페이 가맹점) | 가장 높은 사용률 |
| **토스페이** | `tosspay` | O (토스페이 가맹점) | MZ 선호 |
| 삼성페이 | PG사 통해 지원 | PG사 계약에 포함 | 이니시스/KCP 등 |
| PAYCO | `payco` | O | NHN 결제 |
| SSG페이 | PG사 통해 지원 | 별도 | 신세계 |

### 2.3 네이버페이 연동

**네이버페이 = 2가지 모드**:
1. **결제형**: 우리 사이트 결제창에서 네이버페이 버튼 클릭 → 네이버페이로 결제
2. **주문형**: 네이버 쇼핑에서 "네이버페이 구매" → 우리 서버로 주문 전달

| 항목 | 결제형 | 주문형 |
|------|-------|-------|
| 비유 | 우리 가게에서 네이버페이 카드로 결제 | 네이버 쇼핑몰에서 주문하면 우리에게 알림 |
| PortOne 지원 | O | X (직접 연동) |
| 코드 양 | 최소 (pg: 'naverpay') | 많음 (주문 수신 API) |
| 네이버 리뷰 | X (결제형은 리뷰 불가) | **O (네이버 쇼핑 리뷰 가능)** |
| 추천 시점 | 1단계 (즉시) | 3단계 (네이버 쇼핑 입점 후) |

**핵심**: 네이버 쇼핑에 리뷰가 쌓이려면 "주문형" 네이버페이가 필요. 단순히 결제만 네이버페이로 하면 리뷰가 안 쌓인다.

### 2.4 카카오페이 연동

- PortOne 통합으로 가장 쉽게 추가 가능
- checkout.js에서 `pg: 'kakaopay'`로 호출하면 카카오페이 결제창이 뜸
- 별도 가맹점 계약 필요: [카카오페이 비즈 센터](https://business.kakaopay.com)

### 2.5 토스페이 연동

- 마찬가지로 `pg: 'tosspay'`
- 별도 가맹점 계약: [토스페이먼츠](https://toss.im/tossPay)

### 2.6 코드 수정 사항 (간편결제 공통)

| 파일 | 수정 내용 | 이유 |
|------|----------|------|
| checkout.html | 간편결제 버튼 3개 (네이버/카카오/토스) UI 추가 | 결제 수단 선택 |
| checkout.js | selectPayMethod에 'naverpay', 'kakaopay', 'tosspay' 분기 추가 | 각 간편결제별 pg 파라미터 분기 |
| checkout.js | processCardPayment → processOnlinePayment로 이름 변경 + pg 파라미터 받기 | 카드/간편결제 공통 로직 |
| payment.js | 변경 없음 | PortOne 서버 API는 결제 수단과 무관하게 동일 |

## Part 3: 네이버페이 리뷰 연동

### 3.1 전체 그림 (비유: 백화점 입점)

네이버 쇼핑에 상품이 노출되고, 리뷰가 쌓이려면 3가지가 필요:

```
1. 상품 카탈로그 제출 (EP 피드) = "백화점에 입점 신청서 제출"
   → 네이버가 우리 상품을 검색 결과에 노출

2. 네이버페이 주문형 연동 = "백화점 POS 시스템 연결"
   → 네이버에서 주문하면 우리 시스템으로 전달

3. 주문 완료 처리 = "배송 완료 후 리뷰 요청"
   → 배송 완료 → 네이버가 구매자에게 리뷰 작성 요청
```

### 3.2 네이버 쇼핑 입점 절차

**방법 A: 네이버 쇼핑 EP (상품 피드)**
- "EP" = Entry Point. 우리 상품 정보를 네이버가 읽을 수 있는 형식(XML/TSV)으로 제공
- 비유: "메뉴판을 네이버에 보내는 것". 네이버가 메뉴판을 보고 검색 결과에 노출
- 네이버 검색광고 센터에서 EP 등록 신청

**방법 B: 네이버 스마트스토어**
- 네이버가 제공하는 자체 쇼핑몰에 입점
- 우리처럼 자체 사이트가 있으면 스마트스토어와 **병행** 가능
- 스마트스토어에서 판매한 주문은 자동으로 네이버 리뷰가 쌓임

**방법 A vs B 비교**:

| 항목 | EP (상품 피드) | 스마트스토어 |
|------|--------------|------------|
| 비유 | 우리 가게 위치를 네이버 지도에 등록 | 네이버 백화점 안에 매장 입점 |
| 주문 처리 | 우리 사이트에서 | 네이버에서 |
| 개발 난이도 | 높음 (API 연동) | 낮음 (네이버 관리자에서 직접) |
| 수수료 | 네이버페이 수수료만 | 네이버 수수료 + 매출 연동 수수료 |
| 리뷰 | 네이버페이 주문형 필수 | 자동 |
| 추천 | 장기 (자체몰 중심) | **단기 (빠른 리뷰 축적)** |

### 3.3 EP(상품 피드) 생성 방법

EP는 네이버가 정해놓은 형식으로 상품 정보를 제공하는 파일.
비유: "네이버한테 보내는 상품 카탈로그 엑셀".

**형식**: TSV (탭으로 구분된 텍스트) 또는 XML

**필수 필드**:

| 필드 | 설명 | 우리 DB 매핑 |
|------|------|-------------|
| id | 상품 고유 ID | products.id |
| title | 상품명 | products.name |
| price_pc | PC 판매가 | products.price |
| price_mobile | 모바일 판매가 | products.price (동일) |
| link | 상품 상세 URL | `https://도메인/detail.html?id={id}` |
| image_link | 대표 이미지 URL | products.image |
| category_name | 네이버 카테고리 | 매핑 필요 (스포츠의류 등) |
| naver_category | 네이버 카테고리 코드 | 네이버에서 조회 |
| shipping | 배송비 | 조건부 무료 |

**구현 방법**:
- 서버에 `GET /api/ep/naver` 엔드포인트 추가
- products 테이블에서 공개(published) 상품만 조회
- TSV 형식으로 변환하여 응답
- 네이버 검색광고 센터에 이 URL을 EP 주소로 등록
- 네이버가 주기적(보통 1일 1회)으로 이 URL을 크롤링

### 3.4 네이버페이 주문형 연동

이것이 리뷰 연동의 **핵심**. 네이버 쇼핑에서 "네이버페이로 구매" 버튼을 누르면 우리 서버로 주문이 들어오는 구조.

**흐름**:
```
[고객] → 네이버 쇼핑에서 상품 검색
     → "네이버페이로 구매" 클릭
     → 네이버페이 결제
     → [네이버 서버] → 우리 서버에 주문 전달 (API)
     → 우리가 주문 처리 + 배송
     → 배송 완료 API 호출
     → [네이버] → 고객에게 "리뷰 써주세요" 알림
     → 리뷰 작성 → 네이버 쇼핑에 노출
```

**필요한 API (우리 서버에 구현)**:

| API | 역할 | 설명 |
|-----|------|------|
| 상품 연동 API | 상품 정보 제공 | EP 피드 + 재고/가격 실시간 조회 |
| 주문 수신 API | 주문 받기 | 네이버에서 결제 완료 → 우리 서버로 전달 |
| 주문 확인 API | 주문 승인 | 재고 확인 후 주문 수락/거절 |
| 발주 확인 API | 발주 처리 | 상품 준비 시작 알림 |
| 배송 시작 API | 송장 등록 | 택배사 + 운송장번호 전달 |
| 배송 완료 API | 배송 완료 | 배송 완료 처리 → 리뷰 요청 트리거 |
| 환불 API | 취소/반품 | 환불 처리 |

**네이버 커머스 API 문서**: https://developer.pay.naver.com

### 3.5 리뷰 수집 → 자체 사이트 표시

네이버 쇼핑에 쌓인 리뷰를 우리 사이트에도 보여주는 방법:

| 방법 | 설명 | 가능 여부 |
|------|------|----------|
| 네이버 커머스 API | 리뷰 조회 API로 가져오기 | O (네이버페이 가맹점만) |
| 크롤링 | 네이버 쇼핑 페이지 스크래핑 | X (약관 위반) |
| 위젯 | 네이버 제공 리뷰 위젯 임베드 | O (간편, 추천) |

**추천 방식: 리뷰 위젯 + API 병행**

1. detail.html의 리뷰 탭에 "네이버 쇼핑 리뷰" 섹션 추가
2. 네이버 리뷰 위젯 임베드 (가장 간단)
3. (선택) 주기적으로 API로 리뷰 수집 → 자체 DB에 저장 → 통합 표시

### 3.6 네이버 스마트스토어와의 관계

| 항목 | 자체몰 + 네이버페이 주문형 | 스마트스토어 |
|------|--------------------------|------------|
| 주문 관리 | 우리 관리자 페이지 | 네이버 셀러 오피스 |
| 재고 동기화 | 직접 구현 필요 | 네이버가 관리 |
| 커스텀 기능 | 자유 (AI 디자인 랩 등) | 제한적 |
| 리뷰 | 네이버페이 주문만 | 모든 주문 |
| 추천 | 장기 목표 | **단기 병행 운영 추천** |

**전략 추천**:
1. 단기: 스마트스토어에 기성품(인기 상품) 등록 → 빠른 리뷰 축적
2. 중기: 자체몰에 네이버페이 결제형(PortOne) 연동 → 우리 사이트 결제
3. 장기: 자체몰에 네이버페이 주문형 직접 연동 → 자체몰 주문도 네이버 리뷰

## 주의사항 (developer용)

1. **PortOne V1 vs V2**: 현재 코드는 V1(iamport.js) 기반. V2(portone.js)로 전환하면 SDK가 다르므로 checkout.js 전면 수정 필요. 당장은 V1 유지 추천.

2. **가상계좌 웹훅 보안**: 웹훅 엔드포인트는 PortOne만 호출해야 한다. IP 화이트리스트 또는 서명 검증 필수.

3. **네이버페이 주문형**: 네이버 커머스 API는 인증 방식이 복잡(OAuth2 + HMAC). 관련 문서를 꼼꼼히 읽어야 한다.

4. **재고 동기화**: 네이버 쇼핑과 자체몰에서 동시 판매하면 재고가 꼬일 수 있다. 재고 차감 로직을 중앙에서 관리해야 한다.

5. **테스트 결제 금액**: PortOne 테스트 모드에서는 1,004원 이상의 금액으로 테스트해야 정상 동작한다.

6. **환불 처리**: 카드 결제 환불은 PortOne API로 취소 요청. 부분 환불도 가능하나 PG사마다 정책이 다르다.
